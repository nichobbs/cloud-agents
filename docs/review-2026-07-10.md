# Comprehensive review — 2026-07-10

Scope: full pass over `src/`, `tests/`, `vendor/lyric-docker`, `frontend/src`,
`docker/`, `deploy/`, and `scripts/` looking for bugs, security issues,
optimisations and enhancements. Complements `docs/review-2026-07-03.md`
(backend security posture) and `docs/review-2026-07-03-followup.md`
(frontend/ops/docs) — findings already tracked there are not repeated except
where this pass fixed one.

Items marked **(fixed here)** ship on this PR. The local environment has no
`lyric`/`dotnet` toolchain (the compiler release download is blocked by the
session's egress policy), so backend changes were desk-checked against
`docs/lyric/*` and validated by CI (`build-full.sh`, `verify.sh`,
`lyric test`); the frontend was typechecked and built locally
(`tsc --noEmit`, `vite build`).

## Bugs

1. **(fixed here) Coloured output was silently dropped from the stream.**
   `CloudAgents.Streaming.jsonEscape` escaped `"` `\` `\n` `\r` `\t` but left
   other control characters raw. JSON forbids raw U+0000–U+001F inside string
   literals, and the browser's `JSON.parse` rejects them — and
   `frontend/src/lib/api.ts` catches that parse error and drops the frame as
   malformed. ESC (0x1B) is exactly such a byte, so **every ANSI-coloured log
   line vanished** from the transcript, despite the whole ansi_up rendering
   pipeline existing for it. Control chars are now emitted as `\u00XX`
   (`src/streaming/streaming.l`), with regression tests that build the ESC
   byte via FFI since Lyric literals can't express it.

2. **(fixed here) Multi-line prompts broke container creation.**
   `vendor/lyric-docker`'s `jsonEscapeValue` escaped quotes and backslashes
   but not newlines, so a `PROMPT` env value containing a newline produced an
   unparseable `POST /containers/create` body. Any multi-line message failed
   the whole run. Now escapes `\n` `\r` `\t` too.

3. **(fixed here) JSON injection into the session store.**
   `agentSessionToJson` concatenated raw field values into JSON. A `"` or `\`
   in any user-controlled field (`repoUrl`, `branch`, `model`, …) corrupted
   the entire store on the next save — or let a crafted value forge sibling
   fields (e.g. a `repoUrl` of `https://x","containerId":"...` re-parses with
   an attacker-chosen `containerId`). Fields are now escaped on write and
   properly unescaped on read. Two adjacent parser bugs fixed at the same
   time: the closing-quote scanner used a look-behind check that misread a
   value ending in an escaped backslash (`"a\\"`), and the object-boundary
   scanner counted `{`/`}` inside string values, so a brace in a value
   desynced object splitting. Round-trip tests cover all three.

4. **(fixed here) `POST /api/sessions/{id}/todos` was registered twice** in
   `src/main.l`.

5. **(fixed here) Deleting a session could orphan its container.**
   `deleteSession` removed the session record *before* terminating the
   container; if termination failed the record was already gone, leaving a
   running container with no handle to retry against. Now terminates first
   and only drops the record on success (the 500 tells the caller to retry).

6. **(fixed here) Invalid port crashed the server at startup.** A
   non-numeric or out-of-range `--port`/env value flowed into the Web.Server
   config, whose range check panics. Now validated in `main`, with a warning
   and fallback to 8080.

7. **(fixed here) Frontend sent a `sessionId` field the backend's
   `AddCommentRequest` doesn't declare.** The server derives the owning
   session from the stored message; the stray field was dead weight at best,
   a strict-deserialiser break at worst. Removed through
   `api.ts`/`CommentThread.tsx`/`MessageBlock.tsx`.

## Security

8. **(fixed here) Input hardening at the session boundary**
   (`src/handlers/sessions.l`):
   - `branch` must be a plausible git ref: rejects leading `-` (option
     injection into `git clone --branch`), `..`, `@{`, `\` `"` `~` `^` `:`
     `?` `*` `[`, whitespace, control chars, trailing `/` or `.lock`,
     length > 255.
   - `repoUrl`: still must be `https://`, now also rejects quotes,
     backslashes, spaces, control chars, length > 2048.
   - `model` (create + update): rejects whitespace, quotes, backslashes,
     braces, control chars, length > 128.
   - `harness` must be one of `claude`/`codex`/`opencode` (previously an
     unknown harness silently ran the claude image while storing the bogus
     name).
   These complement — not replace — the store-level escaping in (3):
   validate at the boundary, escape at the sink.

9. **(fixed here) Todo anchors are now verified.**
   `POST /api/sessions/{id}/todos` accepted any `messageId` without checking
   it exists or belongs to that session, so todos could point at other
   sessions' transcripts. Non-empty anchors are now resolved via
   `getMessageSessionId` and must match the target session.

10. **Not fixed — no authentication on any route.** Already the headline of
    `docs/review-2026-07-03.md`; `CloudAgents.Auth` has tested helpers
    (bearer extraction, whitelist, cache expiry) but nothing calls them.
    Wiring real GitHub OAuth validation needs an HTTP client call in the
    request path and middleware support — Phase 3 work, not a drive-by fix.

## New functionality added

11. **(added here) `GET /api/sessions`** — the server-side session list that
    `docs/review-2026-07-03-followup.md` recommended (its §4). The response
    is a trimmed public projection (`sessionId`/`repoUrl`/`branch`/`harness`/
    `model`) built with the store's escaping helper — internal handles
    (`containerId`, `nativeSessionId`) are deliberately excluded since this
    is the first bulk-read surface for session data (review finding #165).
    The frontend now hydrates `SessionsContext` from it once on load and
    merges any sessions missing from `localStorage` (local entries win;
    failures fall back to the local list silently, so an older backend still
    works). Clearing browser storage no longer strands sessions.

## Known issues deliberately not addressed here

- **The env-var session store** (`CLOUD_AGENTS_SESSIONS_JSON`) is still
  non-persistent across restarts and racy under concurrent writes — already
  tracked. The right fix is moving sessions into SQLite; the schema and
  owner-scoped SQL already exist unused in `CloudAgents.Db`
  (`sessionsSchemaSql`, `insertSessionSql`, …). That is the single highest-
  value next change, but it swaps the storage engine under everything and
  deserves its own PR with runtime verification.
- **No real streaming** — output arrives all at once after the container
  exits (headline of the 2026-07-03 follow-up). Needs
  `GET /containers/{id}/logs?follow=true` piped through SSE.
- **`nextSeq` is count-based** (`SELECT COUNT(*)`), so two concurrent
  `addMessage` calls can mint the same `seq`. Harmless while ids are UUIDs
  and writes are effectively serialised by the synchronous handler, but worth
  an `INSERT`-time `MAX(seq)+1` (or integer column) when sessions move to
  SQLite.
- **`GITHUB_TOKEN` is still never injected** into runner containers
  (follow-up review §1), so the GitHub MCP server in every container can't
  authenticate.
- **`extractBearerToken` is case-sensitive** on the `Bearer ` scheme; RFC
  7235 says schemes are case-insensitive. Cosmetic until auth is enforced.
- **`parseCommandLine` splits on bare spaces** — quoted arguments (or an
  executable path containing spaces, which shifts `idx = 1`) parse wrong.
  Fine for `--port N`, worth replacing with `Environment.GetCommandLineArgs`
  if more flags appear.
- **Duplicated FFI array helpers** (`appendString`, `createEmptyStringArray`,
  `indexOfFrom`, JSON parsing) exist in four packages. Consolidating them
  into one shared package is desirable but touches cross-package codegen,
  which has been the epicentre of the upstream compiler bugs — best done as
  an isolated PR once the toolchain has been stable for a while.
- **`CloudAgents.Sqlite` opens a fresh connection per statement** and
  `reader.Close()` isn't in a `defer`. Fine for SQLite at this scale; the
  per-call `SetProvider` try/catch is also redundant after the first call.

## Suggested next steps (in priority order)

1. Move the session store from the env var into SQLite using the existing
   `CloudAgents.Db` schema/SQL — fixes persistence, races, and unlocks the
   status state machine (`SessionStatus`/`nextStatus` currently have zero
   production callers).
2. Real streaming via Docker's follow-logs API.
3. Wire `GITHUB_TOKEN` into `docker_manager.l`'s env list.
4. Enforce auth using the existing `CloudAgents.Auth` helpers.
5. `GET /api/sessions/{id}` (single fetch) to complement the new list
   endpoint, and message pagination for long transcripts.
6. A cancel endpoint for in-flight runs (the 30-minute `taskWaitMs` cap is
   currently the only way out), plus `docker ps` reconciliation at startup.
