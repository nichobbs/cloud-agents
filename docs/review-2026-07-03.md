# Repo review — 2026-07-03

Scope: dependency/build system overhaul (implemented in this change) plus a
broader read-through of `src/`, `deploy/`, `docker/`, `frontend/`, and CI for
bugs, security issues, and incomplete functionality. Items marked **(fixed
here)** were safe to change without a Lyric toolchain to compile-check
against; everything else is a recommendation for a follow-up change, since
this repo has no `lyric`/`dotnet` available in this environment to verify
larger edits against.

## Dependencies (implemented in this change)

- `lyric.toml` previously pulled `Lyric.Web`, `Lyric.Docker`, and
  `Std.Logging` via `path = "../lyric-lang/..."`, i.e. a sibling git checkout
  that isn't part of this repo and isn't available in CI images or the
  `deploy/api.Dockerfile` build context — that Dockerfile could not actually
  have built as checked in. `Lyric.Web`, `Std.Logging`, and
  `Microsoft.Data.Sqlite` are now `[nuget]` binary references (bumped to the
  real latest published versions, `0.4.10`/`0.4.10`/`10.0.9` — confirmed
  against nuget.org's flat-container API, not guessed).
- **`Lyric.Docker` deliberately stayed vendored, and this was verified, not
  assumed.** The first version of this change moved it to NuGet too (a
  guessed, nonexistent `1.3.0`, which failed CI restore outright). After
  pinning the real published `0.4.10`, a follow-up check — downloading the
  actual `.nupkg` from nuget.org and inspecting `Docker.dll`'s embedded
  contract metadata — showed the published package requires an explicit
  `client: DockerClient` argument on every call and has **no
  `waitContainer` function at all**. `src/docker_manager.l` calls
  `createContainer`/`startContainer`/`stopContainer`/`waitContainer`/
  `removeContainer`/`getContainerLogs` with no client argument, so the
  published package is a hard incompatibility, not a nice-to-have upgrade.
  `vendor/lyric-docker` (a local fork with the container-lifecycle surface
  this project needs, "pending upstreaming" per its own comments) is back,
  but declared as an ordinary local package in `[project.packages]` —
  `"Lyric.Docker" = "vendor/lyric-docker/src/docker.l"` — not a separate
  dependency, so there's still no restore-path workaround, no patching, no
  sibling checkout, no inlining dance. Only the parts of the original
  vendoring that were genuinely load-bearing survived; the machinery that
  existed purely to route around a Lyric 0.4.5 restore bug did not.
- `patches/` (two `.patch` files applied to a sibling `lyric-stdlib`/
  `lyric-auth` checkout at build time, for a *different* compiler bug than
  the Docker API gap above) is still removed — nothing in this project's own
  source touches the packages those patches targeted.
- `scripts/build-full.sh` and `scripts/run-api.sh` are now `lyric restore &&
  lyric build`, full stop — no lyric-lang clone, no `sed`/`git apply`
  patching, no inline C# reflection diagnostics.
- `lyric.lock` deleted — it only ever tracked the path-based `[dependencies]`
  entries; with nothing left in that section there's nothing to lock (NuGet
  restore has its own state).
- **`lyric test` (`cmdTestManifest`) crashes with an unhandled
  `System.InvalidCastException`** on this project's manifest — confirmed in
  CI against the real 0.4.10 compiler, not a guess. Checking this repo's own
  git history shows `lyric test` was never actually exercised successfully
  before this change either: prior versions of `scripts/verify.sh` already
  avoided it, compiling a hand-rolled `main()`-with-assertions harness and
  running it via `lyric build && lyric run` instead. `scripts/verify.sh` and
  `deploy/api.Dockerfile` both do that again now (see `docs/BUILD.md`
  "Running tests"). The `tests/*.l` `@test_module` files remain useful as
  documentation of intended behavior but currently cannot be executed by any
  known-working command.

## Security

1. **Critical — no authentication is enforced on any endpoint.**
   `src/handlers/auth.l` fully implements the pure pieces of GitHub-OAuth
   auth (bearer extraction, whitelist check, cache-expiry math, ownership
   check) and has a matching test module, but nothing in
   `src/handlers/sessions.l`, `src/handlers/interactions.l`, or `src/main.l`
   ever calls it. Every route — including `POST /api/sessions` (spins up a
   Docker container that clones an arbitrary `https://` URL and runs an
   arbitrary prompt through the agent CLI) and `DELETE /api/sessions/{id}`
   — is reachable by anyone who can reach the API. `docs/phase3-multi-tenancy.md`
   describes the intended middleware; it was never wired in. The frontend
   already sends an `Authorization` bearer header
   (`frontend/src/lib/api.ts`), so the client side is ready for this.
   Recommended fix: **not** a single cross-cutting aspect — per
   `docs/lyric/gotchas.md` §Aspects, aspects are package-private by default
   and only weave over functions in the *same* package, but the handlers
   live in two different packages (`CloudAgents.Handlers` in `sessions.l`,
   `CloudAgents.Interactions` in `interactions.l`). Either declare one
   `aspect` per package (duplicated `matches:`/`around` blocks in both), or
   skip aspects and have each handler call a shared
   `CloudAgents.Auth.requireUser(headers): Result[GitHubUser, Web.ApiError]`
   helper explicitly as its first line — more boilerplate per handler, but
   works across packages without duplicating the aspect itself and keeps the
   auth check visible at each call site.
2. **Critical — the shared runner credential volume is not per-user.**
   `src/docker_manager.l:39` hardcodes `homeVolume = "claude-home-default"`
   for every session ("Phase 1 default, Phase 3 user-specific" — Phase 3
   never landed). `docker/Dockerfile` mounts that volume's contents at
   `/home/claude-user`, which holds the Claude Code CLI's authenticated
   credentials. Combined with finding 1, any unauthenticated caller can run
   an arbitrary prompt in a container that has read access to those shared
   credentials. `CloudAgents.Db.homeVolumeName(githubUserId, ...)` already
   exists and is unused — this is a wiring gap, not a missing design.
3. **High — session store is a process-env-var blob, not the SQLite table
   already designed for it.** `src/sessions/session_manager.l` reads/writes
   the entire session list as one JSON string in
   `CLOUD_AGENTS_SESSIONS_JSON` via `Environment.Get/SetEnvironmentVariable`.
   Every mutation (`addSession`, `updateSessionContainer`,
   `updateSessionModel`, `removeSession`) is a read-modify-write with no
   locking, so concurrent requests race and can silently drop each other's
   writes; state also does not survive a process restart, contradicting
   ADR-002. Meanwhile `src/db/db_client.l` already defines a fully
   parameterized, `github_user_id`-scoped `sessions` table and SQL builders
   (`insertSessionSql`, `selectSessionByIdSql`, `listSessionsByUserSql`,
   `updateSessionStatusSql`, `deleteSessionSql`, `recoverDanglingSessionsSql`)
   that are exercised only by `tests/db_tests.l` — `initSchema()` in
   `src/db/repository.l` never creates the `sessions` table, and nothing
   calls any of these builders. This is tracked as "pending" in
   `docs/PROGRESS.md` under Phase 2, but is worth calling out as a
   correctness bug (silent session loss under concurrent load), not just a
   missing feature.
4. **Medium — supply-chain: unpinned installer script.** `deploy/api.Dockerfile`,
   `docker/Dockerfile`, and `.github/workflows/ci.yml` all
   `curl https://raw.githubusercontent.com/nichobbs/lyric-lang/main/scripts/install.sh | sh`
   — an unpinned `main`-branch script with no checksum, run on every CI
   build and every prod image build. Consider pinning to a release tag/SHA.
5. **Medium — GitHub token kept in `localStorage`.**
   `frontend/src/lib/api.ts` stores the bearer token in `localStorage`
   rather than memory or an httpOnly cookie, so any future XSS gets
   persistent token theft.
6. **Low — `docker/entrypoint.sh` git-clone argument hygiene.** `REPO_URL`
   is passed to `git clone` as a bare positional argument; it's constrained
   to `https://...` by `isHttpsUrl()` in `src/handlers/sessions.l` today, so
   there's no live exploit, but adding a `--` separator before positional
   args would remove the dependency on that upstream check holding forever.
7. **Low — encrypted-credential-upload design is unimplemented.**
   `docs/phase3-multi-tenancy.md` §3 describes `PUT /api/users/me/credentials`
   with server-side encryption; no such endpoint or encryption code exists
   yet, and `deploy/backup.sh` therefore backs up the shared credential
   volume in plaintext.

## Architecture / dead code

- **(fixed here)** `src/docker_manager.l` declared `taskWaitUnit`, an extern
  `Task.Wait` binding with no callers (only `taskWaitMs` is used) — removed.
- `src/db/db_client.l`'s `sessions` table SQL (`insertSessionSql` through
  `recoverDanglingSessionsSql`, plus `sessionsSchemaSql`) is dead code from
  the live server's point of view — only referenced from tests. Either wire
  it in (see finding 3) or, if the env-var store is intentionally staying
  short-term, note that explicitly so the next reader doesn't assume it's
  connected.
- Route handlers in `sessions.l`/`interactions.l` carry `@post`/`@get`/`@delete`
  annotations (e.g. `@post("/api/sessions")` on `createSession`), but
  `src/main.l` *also* registers every route explicitly by string name via
  `Web.addPost(router, "/api/sessions", "CloudAgents.Handlers.createSession")`.
  Worth confirming with `lyric doc`/the Web library docs whether the
  annotations do anything at all in the current Web version — if they're
  inert, drop them so there's a single source of truth for routing; if
  they're live, `main.l`'s manual registration is the dead one.
- `vendor/lyric-docker/README.md` (removed with the rest of `vendor/`) had
  drifted from the actual API surface — it documented `makeDockerClient()`,
  `systemInfo()`, `listContainers()`, `listImages()` and
  `makeRootlessDockerClient()`, none of which exist in the current
  `docker.l` (`git log` shows they were deleted as dead code in "public API
  simplification"). No action needed now that the file is gone, but a
  signal that generated/vendored docs need the same review discipline as
  code.

## Testing gaps

- No test exercises the HTTP handler layer (`sessions.l`, `interactions.l`)
  at all — `tests/` only covers the pure logic layers (`streaming`, `db`
  builders, `auth` helpers). Once auth is wired in, handler-level tests
  (using `@stubbable`/wire-block patterns per `docs/lyric/reference.md`
  §Testing) would catch regressions like finding 1 going forward.
- `docker_manager.l`'s async orchestration (`runSessionMessageAsync`,
  `terminateSessionContainerAsync`) has no test coverage; it's plausible to
  stub `Lyric.Docker`'s functions via a `@stubbable` interface seam instead
  of calling the free functions directly, which would also make this
  testable.
